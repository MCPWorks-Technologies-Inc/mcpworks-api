# Implementation Plan: MCPWorks Containerized Agents

**Branch**: `003-containerized-agents` | **Date**: 2026-03-11 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/003-containerized-agents/spec.md`

## Summary

Add containerized autonomous agents to the existing Functions platform. Agents are long-running Docker containers with dedicated namespaces, cron scheduling, webhook ingress, encrypted persistent state, function locking, AI engine support (BYOAI), and communication channels. Three new internal-only subscription tiers gate access. 18 new MCP tools enable agent management through AI assistants. Built in 4 phases: shell → webhooks+scheduling → state+locking+cloning → AI+communication.

## Technical Context

**Language/Version**: Python 3.11+
**Primary Dependencies**: FastAPI 0.109+, SQLAlchemy 2.0+ (async), Pydantic v2, Docker SDK 7.0+, APScheduler 3.10+, cryptography (AES-256-GCM), httpx, discord.py
**Storage**: PostgreSQL 15+ (primary, via DO Managed Database), Redis/Valkey 7+ (rate limiting, sessions)
**Testing**: pytest with async support, Docker SDK mocks
**Target Platform**: Linux server (DigitalOcean droplet, s-4vcpu-8gb)
**Project Type**: Single project (backend API)
**Performance Goals**: Lifecycle ops <30s, schedule jitter <5s, webhook latency <2s, state ops <500ms, 20 concurrent agents
**Constraints**: 8 GB total RAM (~6.8 GB available for agents), 4 vCPU (~3.2 available), single-node Phase 1
**Scale/Scope**: 20-26 agents max (builder spec), internal-only users during pre-commercialization

### Principle II: Token Efficiency & Streaming — PASS
- All API responses follow progressive disclosure (return ID + status, not full objects)
- MCP tools return compact responses (<500 tokens per operation)
- Agent lifecycle operations (create, clone) could use SSE for progress — deferred to Phase 2 scaling

### Principle III: Transaction Safety & Security — PASS
- Agent creation is multi-step (DB insert → Docker create → status update) with rollback on failure
- Envelope encryption (AES-256-GCM) for all secrets: AI API keys, state values, channel credentials
- Agent containers run as non-root, read-only filesystem, dropped capabilities
- Network isolation: agents cannot reach database or Redis
- Agent API keys scoped to own namespace only (write+execute, no admin)
- Docker socket access is the primary security concern — mitigated by strict input validation

### Principle IV: Provider Abstraction & Observability — PASS with NOTE
- Container runtime is Docker-specific (not abstracted) — acceptable for Phase 1 single-node
- structlog integration for all agent operations (create, start, stop, destroy, schedule, webhook)
- AgentRun records provide audit trail for all executions
- Admin health endpoint provides fleet-wide observability
- NOTE: Container runtime abstraction deferred to Phase 2 (Docker Swarm)

### Principle V: API Contracts & Test Coverage — PASS
